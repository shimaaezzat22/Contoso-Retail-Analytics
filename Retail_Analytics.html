What makes Parquet columnar — and why it matters

🗂 Row Store vs Column Store — How Parquet reads data

❌ CSV / Row-oriented store

Row 1: [DateKey=20240101, ProductKey=101, StoreKey=5, SalesAmount=450.00, TotalCost=310.00, ...]
Row 2: [DateKey=20240101, ProductKey=102, StoreKey=5, SalesAmount=220.00, TotalCost=150.00, ...]
Row 3: [DateKey=20240102, ProductKey=101, StoreKey=7, SalesAmount=780.00, TotalCost=530.00, ...]
...

→ Power Query asks for SalesAmount only
→ Must read ALL columns of ALL rows first
→ Reads 19 unnecessary columns per row
→ 13M rows × 19 cols = 247M field reads

✅ Parquet / Column-oriented store

Column: DateKey     → [20240101, 20240101, 20240102, ...]
Column: ProductKey  → [101, 102, 101, ...]
Column: StoreKey    → [5, 5, 7, ...]
Column: SalesAmount → [450.00, 220.00, 780.00, ...]  ← read only this
Column: TotalCost   → [310.00, 150.00, 530.00, ...]
...

→ Power Query asks for SalesAmount only
→ Reads ONLY the SalesAmount column stripe
→ 13M rows × 1 col = 13M field reads (column pruning)
→ 19× less I/O for single-column aggregation

Why Snappy compression — not GZIP or uncompressed

Parquet supports multiple compression codecs. Snappy was chosen over GZIP and Zstandard for a specific reason: Snappy prioritises decompression speed over compression ratio. When Power Query reads a Parquet file from SharePoint, it decompresses column stripes on the fly. GZIP achieves ~40% smaller files but decompresses 5–10× slower than Snappy. For a 13M-row fact table, the decompression time difference is material. Snappy provides 5–10× compression vs CSV while keeping decompression fast enough that network latency dominates, not CPU decompression time.

Parquet limitations and how this project resolves them

Limitation 1 — Type metadata conflicts. Parquet embeds its own type metadata (INT32, DOUBLE, BYTE_ARRAY). Power Query's Parquet reader occasionally misinterprets INT32 as Int64 or MONEY as generic number, silently corrupting currency column formatting. There is no warning — the values are numerically correct but formatted wrong.
Resolution → Schema Dictionary injection. A SchemaDictionary.csv maps every (FileName, ColumnName) pair to its authoritative M type string. Power Query strips all Parquet-inferred types and re-applies from the CSV. Parquet metadata is ignored entirely. See Section 6 for the detailed workflow.
Limitation 2 — Streaming parse failures. The Mashup Engine streams large Parquet files lazily from SharePoint. If the SharePoint connection is interrupted mid-stream (common with 13M-row files over corporate networks), the partial parse crashes with a cryptic error. The file appears to load but the table is truncated or empty.
Resolution → Binary.Buffer() anti-streaming. Every Parquet file is fully downloaded into RAM before parsing begins. Binary.Buffer() forces complete file retrieval first. This trades memory for reliability — the correct trade-off for a 31.5M row model where a mid-parse crash wastes 30+ minutes. See Section 5.
Limitation 3 — NULL column schema mismatches. fact.vReturns uses UNION ALL to combine online and physical return events. Physical returns have NULL CustomerKey; online returns have NULL StoreKey. Pandas reads all-NULL integer columns as float64(NaN). When the first non-NULL integer chunk arrives, ParquetWriter sees a schema mismatch (float64 vs int64) and crashes.
Resolution → Nullable schema locking. _make_nullable_schema() marks every field as nullable=True in the Arrow schema before the first write. All chunks are cast with safe=False, allowing float64(NaN) → int64(null) coercion without error. The locked schema is applied to every subsequent chunk, preventing type-drift crashes across chunks.
Limitation 4 — Full re-export on every run. Parquet is a static file format. There is no native delta or incremental mechanism. Every export run re-processes all 13M rows of fact.vOnlineSales from scratch, taking 30+ minutes. As data volume grows, this cost scales linearly.
Resolution → Planned: watermark-based delta export. See Section 9 for the full incremental refresh plan. The current architecture is designed with this upgrade in mind — the SQL views expose DateKey columns that can serve as watermark boundaries for delta queries.

Pattern 1 of 5

Table.Buffer() — Metadata Caching

The single highest-impact Power Query optimisation for SharePoint-hosted models. One line of code eliminates 21 out of 22 SharePoint API calls per refresh cycle.

The core problem: SharePoint.Files() is not a static local lookup. Every time a Power Query table query references DataSource, the Mashup Engine makes a live API call to SharePoint to list the folder contents. Without Table.Buffer(), 22 separate table queries each call SharePoint independently — 22 API calls, 22 network round-trips, 22 authentication handshakes. This is not just slow; it creates rate-limiting risk on corporate SharePoint tenants.

Pattern 2 of 5

The Pure Function Pattern & Static Analyser Evasion

How fxLoadFactParquet and fxLoadDimParquet are designed to fool the Power BI Service static analyser into granting gateway-free cloud refresh.

The core problem — the static analyser: When you publish a Power BI model to the Service and enable scheduled refresh, the Service runs a static analysis pass over all M queries. It scans every function body for data source references (SharePoint.Files, SQL.Database, etc.). If it finds a data source inside a function body, it classifies that function as a dynamic data source — meaning it cannot know at parse time what the source URL will be. Dynamic sources require an on-premises data gateway. A gateway is infrastructure that must be deployed, maintained, and kept running. Avoiding it is a significant operational simplification.
❌ Naïve approach — data source INSIDE function body → gateway required
-- This is what you SHOULD NOT do:
fxLoadParquet_WRONG = (FileName as text) =>
  let
    Source = SharePoint.Files(pSharedSiteURL),       -- ← DATA SOURCE INSIDE FUNCTION
    File   = Table.SelectRows(Source, each [Name] = FileName),
    ...
  in ...

-- Static analyser sees SharePoint.Files() inside the function body
-- → flags as "dynamic data source"
-- → Power BI Service: "This datasource requires a gateway"
-- → You must install and maintain an on-premises gateway ❌
✅ Pure function pattern — data source OUTSIDE function → no gateway needed

-- Binary passing (BinaryData) → goes into the pure function
-- Static analyser: sees only a buffered table reference → static source → no gateway
Result: Power BI Service classifies the model as using a static data source. Scheduled refresh works directly from the cloud — no gateway infrastructure required. This decision alone eliminates the need to deploy, maintain, and monitor an on-premises gateway server.

⚠️ Maintenance rule — never break the pure function contract

Adding any data source call (SharePoint.Files, SQL.Database, Web.Contents, etc.) inside fxLoadFactParquet or fxLoadDimParquet will silently break cloud refresh. The Service will not warn you at publish time — it will only fail at the first scheduled refresh. The pure function boundary must be treated as a hard architectural rule: all data source navigation lives in named expressions (DataSource, SchemaMapping), never inside loader functions.

Pattern 3 of 5

Binary.Buffer() — Anti-Streaming Pattern

Why fully buffering a 13M-row Parquet file into RAM before parsing is the correct engineering trade-off for a SharePoint-hosted model.

The Mashup Engine's default behaviour: Without Binary.Buffer(), the Power Query Mashup Engine is lazy. When Parquet.Document(BinaryContent) is called, the engine begins streaming the binary content from SharePoint and parsing it incrementally — consuming less RAM at the cost of a persistent, live network connection for the duration of the parse. For small local files, this works fine. For large Parquet files over a corporate SharePoint connection, it fails.

DAX Cross-Fact Pattern

TREATAS — Cross-Fact Table Filtering

How DAX bridges two fact tables that share a business key but have no model relationship, without corrupting the star schema.

The problem TREATAS solves: In a strict Kimball star schema, creating a direct relationship between two fact tables is forbidden. It creates ambiguous filter propagation paths — a "fan trap" — where a single dimension filter propagates to multiple fact tables simultaneously, multiplying rows and producing incorrect aggregations. Yet there are legitimate business questions that require filtering one fact table based on data in another. TREATAS is the DAX solution that enables cross-fact filtering without creating a model relationship.
TREATAS in plain language
TREATAS(table_expression, column1, column2, ...) takes a table (or column values) and tells DAX to treat it as if those values were a filter on the specified target columns — even though no relationship exists between them in the model. The name stands for TREAT AS: treat this set of values as if they were coming from a related column.

Think of it as a temporary virtual relationship that exists only for the duration of the current CALCULATE evaluation, then vanishes. It does not persist in the model and cannot create fan traps.

Improvement Plan · Not Yet Implemented

Incremental Refresh — Watermark & Partition Plan

The current architecture re-processes all 31.5M rows on every export and refresh cycle. This plan introduces two coordinated mechanisms to make both the Python export and the Power BI refresh incremental.

⚠️ This is a plan for your approval — no implementation has been done. Review the concepts and architecture below, then confirm approval to proceed.

Concept 1 — Watermark-Based Delta Export (Python layer)

A watermark column is a monotonically increasing value (typically a date or timestamp) that marks the "high water mark" of the last successfully exported data. On the next export run, only rows where the watermark column exceeds the last stored watermark are exported — the delta. All prior rows are assumed unchanged and already exported.

In this project, DateKey (INT YYYYMMDD) is the natural watermark column for all fact views. It is already present in all 11 fact tables, is INT-typed (fast comparison), and is monotonically increasing (new orders always have higher DateKeys than older orders).

Concept 2 — Power BI Incremental Refresh Partitions

Power BI's incremental refresh (available on Premium capacity) splits a table into multiple time-partitioned segments. The model maintains one partition per time window (e.g., one per month). On each refresh, only the most recent N partitions are refreshed — older partitions are kept as-is in compressed VertiPaq storage, never re-loaded. The RangeStart and RangeEnd parameters define the refresh boundary for each partition.

This requires: (1) Power BI Premium or Premium Per User capacity, (2) a datetime or date column defined as the partition key, (3) RangeStart and RangeEnd Power Query parameters in the M code, (4) the data source to support query folding so Power BI can push the date range filter down to the source.

How to Set Dual Storage Mode — Step-by-Step

Dual mode allows a dimension table to behave as Import when queried with Import fact tables, and as DirectQuery when queried with DirectQuery fact tables. This is the recommended storage mode for all dimension tables in a Composite Model — it eliminates "weak relationships" and ensures full DAX compatibility in both query contexts.

Why Dual instead of plain Import for dimensions in Composite Mode: When a dimension table is Import and a fact table is DirectQuery, Power BI creates a "limited relationship" (also called a weak relationship) between them. Limited relationships lose bidirectional cross-filtering capability and some DAX functions behave differently. Dual mode eliminates limited relationships — the dimension can act as Import (fast) or DirectQuery (live) depending on which fact table it is being queried with, making it fully compatible with both storage modes simultaneously.
